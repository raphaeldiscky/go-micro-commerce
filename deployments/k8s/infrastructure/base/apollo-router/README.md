# Apollo Router Deployment for Kubernetes

This directory contains the Kubernetes deployment configuration for the Apollo Router using the official Apollo Router Helm chart with Federation v2.

## Overview

The GraphQL Gateway uses Apollo Router v2.7.0 to federate 6 microservice subgraphs:

- **auth-service** (port 8081)
- **chat-service** (port 8088)
- **notification-service** (port 8086)
- **order-service** (port 8083)
- **cart-service** (port 8089)
- **payment-service** (port 8084)

## Architecture

### Deployment Method

- **Helm Chart**: Official Apollo Router Helm chart (`apollo/router`)
- **Configuration**: ConfigMaps for router config, Rhai scripts, and supergraph schema
- **Scaling**: HorizontalPodAutoscaler (2-10 replicas based on CPU/Memory)
- **Observability**: OpenTelemetry tracing, Prometheus metrics, Grafana dashboards
- **GitOps**: Schema composition in CI/CD, Kustomize for manifest generation

### ⭐ Industry Best Practice: CI-Based Schema Composition

**Why CI/CD Composition?**

✅ **GitOps Compliant**: Git is the single source of truth
✅ **Fast Deployment**: No runtime composition delays
✅ **Auditable**: Full git history of schema changes
✅ **CI Validation**: Schema errors caught before production
✅ **Deterministic**: Same git commit = same deployment

**Schema Workflow**:

```
Service Schema Change
      ↓
GitHub Actions CI/CD
  - Merge schema fragments
  - Compose development supergraph
  - Compose Kubernetes supergraph  ← NEW
  - Commit to git [skip ci]
      ↓
Git (supergraph-schema.graphql)
      ↓
   ┌──────────────┬──────────────┐
   ↓              ↓              ↓
Tilt (Local)  ArgoCD (Prod)  Kustomize
   ↓              ↓              ↓
 ConfigMap     ConfigMap     ConfigMap
(from file)   (from file)   (generated)
```

### Components

1. **ConfigMaps** (Generated by Kustomize)
   - `apollo-router-config`: Router configuration (CORS, headers, telemetry)
   - `apollo-router-rhai`: Rhai script for Set-Cookie header propagation
   - `apollo-supergraph-schema`: Generated from `supergraph-schema.graphql` file

2. **Helm Chart**
   - `values.yaml`: Apollo Router Helm chart configuration
   - Replica count, resources, health checks, security context

3. **Schema Files** (Committed to Git)
   - `supergraph-schema.graphql`: **CI-composed** supergraph (GitOps source of truth)
   - `supergraph.yaml`: Composition config (K8s service DNS URLs)
   - `schemas/` (deleted): Local copies no longer needed

4. **Observability**
   - Prometheus ServiceMonitor (generated by Helm chart)
   - GraphQL-specific metrics on port 9091
   - Health checks on port 4088
   - OTEL tracing to `otel-collector:4317`

## Deployment

### Prerequisites

1. Kubernetes cluster (Kind, MicroK8s, or any K8s cluster)
2. Helm 3.x installed
3. Kubectl configured
4. All subgraph services deployed and healthy

### Deploy with Tilt (Local Development)

The Apollo Router is automatically deployed when running `tilt up`:

```bash
tilt up
```

**Tilt Workflow** (Dev-Prod Parity):

1. Uses Kustomize to generate ConfigMaps from files
2. Deploys ConfigMaps (router config, Rhai script, **supergraph schema**)
3. Deploys Apollo Router via official Helm chart
4. Waits for all subgraph services to be healthy
5. Port-forwards 4000 (GraphQL) and 9091 (metrics)

**Key Point**: Tilt uses the **same Kustomize configuration** as production (ArgoCD), ensuring dev-prod parity.

### Deploy with ArgoCD (Production)

Apollo Router is deployed via ArgoCD GitOps workflow:

1. **Schema composed in CI/CD** (.github/workflows/backend.yml)
2. **Committed to git** (supergraph-schema.graphql)
3. **ArgoCD syncs** Kustomize manifests
4. **ConfigMap generated** from file by Kustomize
5. **Apollo Router deploys** with new schema

See `deployments/k8s/README.md` for ArgoCD ApplicationSet configuration.

### Manual Deployment (Not Recommended)

⚠️ **For emergency use only** - violates GitOps principles:

```bash
# Build Kustomize manifests
cd deployments/k8s/infrastructure/apollo-router
kustomize build . | kubectl apply -f -

# Or apply with kubectl kustomize
kubectl apply -k .
```

**Why Kustomize?**

- Generates `apollo-supergraph-schema` ConfigMap from `supergraph-schema.graphql` file
- Same approach for local dev (Tilt) and production (ArgoCD)
- No manual ConfigMap generation needed

## Schema Composition

### CI/CD-Based Workflow

Supergraph schema composition now happens **automatically in CI/CD** when service schemas change.

**Why This Approach?**

- ✅ **GitOps Compliant**: Schema changes tracked in git history
- ✅ **CI Validation**: Schema composition errors caught before production
- ✅ **Fast Deployment**: No runtime composition delays
- ✅ **Auditable**: Full audit trail via git commits
- ✅ **Deterministic**: Same commit always produces same schema

### Automatic Schema Update Workflow

```
1. Developer changes service schema
   Example: auth-service/graph/schema/user.graphql

2. Commit and push to GitHub
   git add auth-service/graph/
   git commit -m "feat(auth): add email verification field"
   git push

3. GitHub Actions CI/CD runs (.github/workflows/backend.yml)
   ├─ Merge schema fragments (scripts/merge-graphql-schemas.sh)
   ├─ Compose development supergraph (graphql-gateway/)
   ├─ Compose Kubernetes supergraph (deployments/k8s/infrastructure/apollo-router/)
   └─ Commit schemas back to git [skip ci]

4. Git receives new commit
   Files updated:
   - graphql-gateway/supergraph-schema.graphql (development)
   - deployments/k8s/infrastructure/apollo-router/supergraph-schema.graphql (K8s)

5. Deployment picks up changes
   - Tilt (local): Detects file change, rebuilds Kustomize, updates ConfigMap
   - ArgoCD (prod): Detects git change, syncs new schema, restarts router
```

### Manual Schema Composition (Development/Testing)

If you need to compose the schema locally without CI:

```bash
# Prerequisites: Install Rover CLI
curl -sSL https://rover.apollo.dev/nix/latest | sh

# Compose Kubernetes supergraph locally
cd deployments/k8s/infrastructure/apollo-router
rover supergraph compose --config supergraph.yaml > supergraph-schema.graphql

# Test with Kustomize
kustomize build . | grep -A 20 "apollo-supergraph-schema"

# Commit if needed
git add supergraph-schema.graphql
git commit -m "chore(graphql): update K8s supergraph schema"
```

## Configuration

### Router Configuration

Edit `configmap-router.yaml` to update:

- CORS settings
- Header propagation rules
- OpenTelemetry configuration
- Rate limiting
- Timeouts

After changes:

```bash
kubectl apply -f configmap-router.yaml
kubectl rollout restart deployment apollo-router
```

### Helm Values

Edit `values.yaml` to update:

- Replica count
- Resource limits
- Health check settings
- Security context
- Environment variables

After changes:

```bash
helm upgrade apollo-router apollo/router \
  --values values.yaml \
  --namespace default
```

### Subgraph URLs

Subgraph routing URLs are configured in `supergraph.yaml` using Kubernetes Service DNS:

```yaml
subgraphs:
  auth-service:
    routing_url: http://auth-service:8081/graph
```

Update `supergraph.yaml` and re-run schema composition if URLs change.

## Observability

### Metrics

Apollo Router exposes Prometheus metrics on port 9091:

```bash
# Port-forward metrics endpoint
kubectl port-forward svc/apollo-router-metrics 9091:9091

# Access metrics
curl http://localhost:9091/metrics
```

**Key Metrics:**

- `apollo_router_http_requests_total` - Total HTTP requests
- `apollo_router_http_request_duration_seconds` - Request latency
- `apollo_router_graphql_error` - GraphQL errors
- `apollo_router_cache_hit_count` - Cache hit rate

### Health Checks

Health endpoint on port 4088:

```bash
# Port-forward health endpoint
kubectl port-forward deployment/apollo-router 4088:4088

# Check health
curl http://localhost:4088/health
```

### Tracing

Apollo Router sends OpenTelemetry traces to `otel-collector:4317`:

- View traces in Grafana Tempo
- Access Grafana at http://localhost:3000

### Logs

```bash
# View Apollo Router logs
kubectl logs -f deployment/apollo-router

# Filter for specific log levels
kubectl logs -f deployment/apollo-router | grep ERROR
```

## Troubleshooting

### Router Not Starting

1. Check ConfigMaps exist:

```bash
kubectl get configmaps | grep apollo
```

2. Check pod logs:

```bash
kubectl logs deployment/apollo-router
```

3. Verify subgraph services are healthy:

```bash
kubectl get pods | grep -E 'auth|chat|notification|order|cart|payment'
```

### Schema Validation Errors

If schema composition fails in CI/CD:

1. Check GitHub Actions workflow logs
2. Review schema compatibility errors
3. Test locally:

```bash
cd deployments/k8s/infrastructure/apollo-router
rover supergraph compose --config supergraph.yaml
```

4. Fix service schemas and retry

### Subgraph Unreachable

1. Check service DNS resolution:

```bash
kubectl run -it --rm debug --image=curlimages/curl --restart=Never -- \
  curl http://auth-service:8081/graph
```

2. Verify subgraph health:

```bash
kubectl exec -it deployment/apollo-router -- \
  wget -O- http://auth-service:8081/health
```

### High Latency

1. Check resource usage:

```bash
kubectl top pods -l app.kubernetes.io/name=router
```

2. Scale replicas:

```bash
kubectl scale deployment apollo-router --replicas=5
```

3. Check HPA status:

```bash
kubectl get hpa
```

## Scaling

### Manual Scaling

```bash
# Scale to 5 replicas
kubectl scale deployment apollo-router --replicas=5
```

### Auto-Scaling (HPA)

HPA is configured in `values.yaml`:

- **Min replicas**: 2
- **Max replicas**: 10
- **CPU threshold**: 70%
- **Memory threshold**: 80%

Adjust in `values.yaml` and upgrade Helm chart.

## Security

### Non-Root User

Apollo Router runs as non-root user (UID 1000) with:

- Read-only root filesystem
- Dropped capabilities
- Security context constraints

### Network Policies

Consider implementing network policies to:

- Restrict egress to subgraph services only
- Allow ingress from Traefik only
- Block lateral movement

## Ports

- **4000**: GraphQL endpoint (HTTP)
- **4088**: Health check endpoint
- **9091**: Prometheus metrics

## Dependencies

Apollo Router requires:

- All 6 subgraph services running
- `otel-collector` for tracing
- `redis-cluster` (optional, for caching)
- `kafka-cluster` (used by subgraphs)
